### 热力图二次深度修复（2026-04-26 15:40）

**真正根因（v-if/finally 时序 bug）**：
- `loadHeatmap()` 原来在 `try` 块里 `await nextTick()×2` 再 `buildHeatmap()`，但此时 `heatmapLoading` 还是 `true`（finally 还没执行）
- 所以 `v-if="heatmapLoading"` 依然显示"加载中..."，`v-else` 分支的 `heatmapRef` 根本没挂载，echarts.init 拿到的是 null
- 修复方案：将 `heatmapLoading = false` 移到 finally 先执行，再在 finally 之后（函数尾部）await nextTick×2 后调用 buildHeatmap
- 同步修复：`buildHeatmap` 改为构建完整的 24×n 格子（缺失小时填 0），而不是只放实际有数据的 5 行，确保热力图网格完整显示
- 后端 `visual_service.py` 将 `Decimal` 类型 avg_hot_value 转为 `float`，防止 JSON 序列化异常

### 三个前后端 Bug 修复（2026-04-26 15:30）

**问题一：热搜活跃时间热力图空白**
- 根因：ECharts heatmap `series.data` 格式为 `[xIndex, yIndex, value]`，但旧代码传入的是 `[d.hour, dateIndex, count]`，当 xAxis 是 category 类型（`["0时","1时"...]`）时，`d.hour` 是数字但 ECharts category 轴需要的是数组下标（0-23 与小时数字相同，但需确保），同时 `heatmapRef` 在 v-else 条件分支中，单个 nextTick 不够，DOM 尚未完成挂载导致 echarts.init 失败。
- 修复：`Dashboard.vue` 的 `buildHeatmap()` 完全重写：销毁并重新 init 实例、使用 `dates.sort()` 排序、tooltip 修正为正确解包 `[hIdx, dIdx, cnt]`；`loadHeatmap()` 改为双重 `nextTick`。

**问题二：爆发趋势 Top20 全部 98%**
- 根因：`_rule_burst_probability` 中 `rank_change_score = (rank_change + 5.0) / 10.0`，大量早期数据 rank_change ≥ 5 时该分数为 1.0，导致所有概率被推高到 0.95 附近；模型预测路径 clip 范围 0.02-0.98 太宽，不足 3 次的数据也获得了高概率。
- 修复：`ml/predict_job.py` 重写 `_rule_burst_probability()`：改用符合信号语义的映射（rank_change 上升=低数字=高分）、5 特征加权、出现次数<3 限制上限 0.55；`_build_prediction_df()` 的模型路径加入 appear_count<3 惩罚；clip 收紧到 0.02-0.96。

**问题三：关键词搜索无响应**
- 根因：`trendText` / `getTrendClass` 只识别英文 rising/stable/falling，对任何其他值都返回默认值（显示正常但视觉上无区分）；`handleSearch` 在 keyword 为空时设置 searched=false 后直接 return，搜索结果区域不刷新；后端 `/api/ml/burst/search` 接口可能因为表空而返回 `[]` 但前端没有明确提示。
- 修复：`BurstAnalysis.vue` 增强 trendText/getTrendClass 兼容中英文、handleSearch 改为先设 searched=true 再返回错误提示、搜索结果表扩充至 10 列（含热度变化率/排名变化/出现次数）、增加搜索数量提示条；`ml_analysis_service.get_burst_by_keyword` 用 LEFT JOIN 关联特征表补充 `hot_value_change`/`feature_date`/`raw_probability`；`backend/routes/analysis.py` 新增 `/api/analysis/burst/keyword`、`/api/analysis/burst/top`、`/api/analysis/hourly-heatmap` 三个别名路由；`frontend/src/api/index.js` 新增 `getBurstByKeyword` 和 `getAnalysisHourlyHeatmap`。

**修改文件**：Dashboard.vue、BurstAnalysis.vue、ml/predict_job.py、backend/routes/analysis.py、backend/routes/ml_analysis.py、backend/services/ml_analysis_service.py、frontend/src/api/index.js

### SentimentMap.vue 地图颜色修复

**问题**：中国地图各省份颜色几乎一样，没有根据评论人数深浅变化。
**根因**：geoData 中的省份名称是短名称（如"广东"），而 GeoJSON 中的是完整名称（如"广东省"），导致 ECharts map series 无法匹配 data.name 和地图区域 name。

**修改文件**：`frontend/src/views/SentimentMap.vue`

**修改内容**：
1. 新增 `provinceNameMap`：将 34 个省级行政区的短名称映射为完整名称（省/市/自治区/特别行政区）。
2. 新增 `normalizeProvinceName(name)`：若名称已以完整后缀结尾则直接返回，否则从映射表中查找。
3. 新增 `normalizeMapData(data)`：将原始 geoData 转换为标准格式 `{ name, value, rawName }`，其中 name 为标准化后的完整省份名。
4. 修改 `buildMapChart()`：
   - `series[0].data` 改用 `normalizeMapData(geoData.value)`，确保名称与 GeoJSON 匹配。
   - `visualMap` 改为 `piecewise` 分段色，6 个区间（4万以上/3-4万/2-3万/1-2万/1万以下/无数据），颜色从深蓝到浅灰，差异更明显。
   - 移除 `series.itemStyle.areaColor` 固定颜色，避免覆盖 visualMap 着色，仅保留 `borderColor` 和 `borderWidth`。
   - tooltip 保持不变，仍显示省份名和评论数。
5. 右侧「省份评论 Top10」排行榜仍使用原始 `geoData.value`，不显示完整名称，体验不变。

**构建验证**：`npm run dev` 成功启动，服务正常运行。
